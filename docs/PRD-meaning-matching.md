# PRD: 뜻 입력 매칭 UX 개선
**작성일:** 2026-03-17  
**상태:** 승인 후 구현

---

## 1. 문제

현재 뜻 입력 정답 판정이 **문자열 완전 일치** 방식이라 UX가 나쁨.

### 1-1. 실제 발생 케이스
| CSV 정답 | 사용자 입력 | 현재 결과 | 올바른 결과 |
|---------|-----------|---------|-----------|
| `존경하다; 중하게 여기다` | `존경하다, 중하게 여기다` | ❌ | ✅ |
| `존경하다; 중하게 여기다` | `중하게 여기다; 존경하다` | ❌ | ✅ |
| `1. 약속하다 2. 예약하다` | `약속하다 예약하다` | ❌ | ✅ |
| `발생시키다` | `발생시키다 ` | ❌ | ✅ |

### 1-2. 근본 원인
- 구분자 차이 (`;` vs `,` vs 공백)
- 순서 차이
- 번호 prefix 포함 여부
- 앞뒤 공백

---

## 2. 뜻 필드 데이터 패턴 분류

| 패턴 | 예시 | 의미 단위 |
|------|------|---------|
| 단일 | `주다` | 1개 |
| 세미콜론 | `존경하다; 중하게 여기다` | 2개 |
| 번호 | `1. 약속하다 2. 예약하다 3. 고용하다` | 3개 |
| 혼합 | `가능성; 기회` | 2개 |

---

## 3. 설계 결정

### 3-1. 매칭 전략: "의미 단위 분리 후 부분집합 검사"

```
정답: "존경하다; 중하게 여기다"
→ parts: ["존경하다", "중하게 여기다"]

사용자 입력: "존경하다, 중하게 여기다"  
→ normalized: ["존경하다", "중하게 여기다"]

판정: 두 파트 모두 포함 → ✅ 정답
```

### 3-2. 파싱 로직

**정답 파싱 (CSV 값):**
1. 번호 prefix 제거: `1. 약속하다 2. 예약하다` → `["약속하다", "예약하다"]`
2. 세미콜론 분리: `존경하다; 중하게 여기다` → `["존경하다", "중하게 여기다"]`
3. 단일 의미: `주다` → `["주다"]`

**사용자 입력 파싱:**
- 구분자를 통일 (`,`, `;`, 공백+공백, `1.` `2.` 등) 후 단위 추출
- 각 단위 normalize (trim, 공백 collapse)

### 3-3. 판정 기준
- 정답의 모든 의미 단위가 사용자 입력에 **포함**되면 정답
- 순서 무관
- 구분자 무관 (`,` `;` 등 자유롭게 사용 가능)
- 대소문자 무관 (영문 포함 시)
- 앞뒤 공백 무관

### 3-4. UI 피드백 개선
현재: 오답 시 "정답: 존경하다; 중하게 여기다" 텍스트만 표시  
개선:
- ✅ 정답 시: 초록 테두리 + 체크
- ❌ 오답 시: 빨간 테두리 + "빠진 항목" 표시
  ```
  ❌ 입력 오류
  빠진 항목: 없음 (구분자 차이만 있었음 → ✅ 정답 처리됨)
  ```

## 5. 엣지케이스

| 케이스 | 처리 방식 |
|-------|---------|
| 사용자가 일부만 입력 (`약속하다`만) | ❌ 나머지 빠진 항목 표시 |
